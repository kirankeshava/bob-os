## Page layout (single page with sections)
### Global controls (top bar)
- Date range: `Last 7 days` (default), `Last 30 days`, `Custom`.
- Location filter: `All locations` + multi-select.
- Source filter (optional): Google / Yelp / Manual.
- Button: **Export CSV** (downloads `metrics.csv` for current filters).

### Section A — Sync health
**Cards:**
1) `Locations enabled`: count of Location where `gbpSyncEnabled=true` OR `yelpSyncEnabled=true` OR `manualEnabled=true` (depending on your flags).
2) `Locations failing`: count where `lastSyncStatus='error'` OR last error within 24h.
3) `Last sync`: timestamp of most recent successful sync across selected locations.

**Table: Locations health**
Columns:
- Location name
- Source enabled (GBP/Yelp/Manual)
- Last sync at
- Last sync result (OK / Error)
- Last error (truncated)
- Reviews ingested (range)
Actions:
- “View reviews” deep-link to `/app/reviews?locationId=...`

**Computation notes:**
- Use `Location.lastGbpSyncAt`, `Location.lastGbpSyncError` (or whatever fields exist). If not present, fall back to AuditLog events (`action='google_sync_success'|'google_sync_error'`).

## Section B — Activation funnel (ingest → drafted → approved → posted)
**Definitions (must be consistent everywhere):**
- `Ingested`: Review.createdAt within date range.
- `Drafted`: DraftReply.createdAt within date range AND has a Review in range (or simply drafted in range; choose one and keep consistent—recommended: drafted in range regardless of review date).
- `Approved`: DraftReply.approvedAt within date range.
- `Posted`: DraftReply.postedAt within date range OR status in (`posted_manual`, `posted_api`).

**Cards:**
- Ingested reviews
- Drafts generated
- Approved replies
- Posted replies
- Approval rate = approved / drafted
- Posting rate = posted / approved

**Funnel table (by location):**
Location | Ingested | Drafted | Approved | Posted | Median time-to-first-draft | Median time-to-approval | Median time-to-post

**Time metrics definitions:**
- `time-to-first-draft`: DraftReply.createdAt - Review.createdAt (first draft per review).
- `time-to-approval`: DraftReply.approvedAt - DraftReply.createdAt.
- `time-to-post`: DraftReply.postedAt - DraftReply.approvedAt.

**Important exclusions:**
- Exclude rejected drafts from approval/posting denominators.
- If multiple drafts per review: use the latest approved draft for posting; use the first draft for time-to-first-draft.

## Section C — SLA / Negative review handling
**Cards:**
- Negative reviews (range): count where `sentiment='negative' OR rating<=2`.
- Negative share: negative / ingested.
- Alerts fired: AlertEvent.createdAt in range.
- SLA breaches: negative reviews with no approved reply within X hours (X = business config; default 24h).

**Table: Breaches (top 20)**
Review date | Location | Rating | Snippet | Age | Status (no draft / drafted / approved not posted)

**Computation:**
- A negative review is “handled” if exists DraftReply with `approvedAt` within SLA window from Review.createdAt.

## Prisma/SQL aggregation approach (pseudocode)
Assume Postgres.

### Base filters
- reviewsWhere: `businessId`, `createdAt between start/end`, optional `locationId in (...)`, optional `source`.

### Ingested
- `SELECT count(*) FROM Review WHERE <filters>`

### Drafted / Approved / Posted
- drafted: `SELECT count(*) FROM DraftReply WHERE createdAt between start/end AND reviewId IN (SELECT id FROM Review WHERE businessId=... [optional location/source])`
- approved: `WHERE approvedAt between start/end`
- posted: `WHERE postedAt between start/end OR status IN ('posted_manual','posted_api')`

### Median time calculations
- time-to-first-draft:
  - subquery firstDraftPerReview: `SELECT reviewId, MIN(createdAt) AS firstDraftAt FROM DraftReply GROUP BY reviewId`
  - join with Review to compute deltas; median via `percentile_cont(0.5) within group (order by extract(epoch from (firstDraftAt - Review.createdAt))/60)`

### SLA breaches
- negatives: `WHERE sentiment='negative' OR rating<=2`
- breach if not exists approved draft within SLA hours:
  - `NOT EXISTS (SELECT 1 FROM DraftReply dr WHERE dr.reviewId=r.id AND dr.approvedAt IS NOT NULL AND dr.approvedAt <= r.createdAt + interval '24 hours')`

### Themes
- If Review.categoryLabels stored as array/json:
  - explode labels and group by label + sentiment.

## Implementation checklist (fast path)
1) Add `/api/admin/metrics` route handler (server) with zod validation for query params.
2) Implement aggregations with Prisma `$queryRaw` for medians + grouped counts; use Prisma `count()` for simple counts.
3) Build `/app/admin/metrics` page: server component fetch -> render cards + tables; add a simple sparkline optional later.
4) Add CSV route `/api/admin/metrics.csv` (same aggregations, flattened).
5) Add links from dashboard to reviews queue filters (locationId, sentiment=negative, status=needs_approval).

This spec intentionally avoids new paid services, avoids new infra, and uses tables you already have (Review, DraftReply, Location, AlertEvent, AuditLog).
